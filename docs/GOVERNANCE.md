# RadTempCF Governance Document

## Overview

This document outlines the governance structure and decision-making processes for the RadTempCF project.

## Core Values

1. **Openness**: All governance discussions and decisions are public
2. **Inclusivity**: Anyone can contribute regardless of background
3. **Transparency**: Decisions are documented with rationale
4. **Meritocracy**: Influence is earned through contribution
5. **Patient Safety**: Template quality prioritizes patient care

## Roles & Responsibilities

### Maintainers
- Set technical direction
- Approve PRs and releases
- Resolve conflicts
- Manage project infrastructure
- **Current**: @ OWNERS (see CODEOWNERS)

### Contributors
- Submit PRs and issues
- Participate in discussions
- Review code and templates
- Help with documentation
- **Anyone can become a contributor**

### Advisory Board
- Strategic guidance
- Community representation
- Feature roadmap input
- Conflict mediation

### Users
- Use templates in clinical practice
- Provide feedback
- Report issues
- Suggest improvements

## Decision Making

### Authority Levels

| Decision Type | Authority | Example |
|---------------|-----------|---------|
| **Critical** | Maintainers + Advisory | Security, architecture changes |
| **Major** | Maintainers | Feature additions, schema changes |
| **Minor** | Contributors | Documentation, templates, fixes |

### Process

1. **Proposal**: Start a discussion or issue
2. **Feedback**: Gather input from community (5+ days for major)
3. **Decision**: Maintainers make final call
4. **Documentation**: Publish rationale
5. **Implementation**: Assigned to contributor

### Consensus

- For minor: Any maintainer can approve
- For major: Requires 2+ maintainer approval
- For critical: Full maintainer team + advisory board consensus

## Conflict Resolution

### Step 1: Direct Discussion
- Parties attempt to resolve privately
- Focus on facts and project goals

### Step 2: Mediated Discussion
- Involve a neutral maintainer
- Document the disagreement

### Step 3: Formal Vote
- Maintainers vote (simple majority)
- Advisory board breaks ties

### Escalation
- If maintainer is involved, defer to other maintainers
- Final appeal to advisory board chair

## Communication Channels

| Channel | Purpose | Moderation |
|---------|---------|-------------|
| GitHub Issues | Bug reports, feature requests | Maintainers |
| GitHub Discussions | Q&A, ideas, showcase | Community |
| Discord (future) | Real-time discussion | Moderators |
| Email | Private matters | Maintainers |
